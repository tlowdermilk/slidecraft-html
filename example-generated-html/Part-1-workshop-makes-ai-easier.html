Part 01 · The Workshop

How a one-day workshop makes programming with AI actually easier.

In this part

What changes when a team stops prompting and starts shipping. A field tour of the HVE Core kit — agents, skills, instructions, and the Research → Plan → Implement loop — and what it does to your Monday.

Most teams treat AI like a clever intern with a long attention span. A workshop turns it into a kit — a set of agents, skills, and rules your repo already knows how to use.

Field notes · Hypervelocity Engineering · 2026

01 The Problem

AI without scaffolding is improv.

Pasting a prompt into a chat window gets you something. It rarely gets you something your team can ship — something that matches the repo's conventions, respects the security boundaries, names variables the way the rest of the codebase does, and survives Monday's code review.

The default failure mode is not that the model is wrong. It's that the model is uncalibrated. It doesn't know which folder your tests live in, which auth pattern your tenant approved, which agent on your team owns the deploy, or that your handler.ts already has three helpers that do almost-what-it-just-invented.

A workshop's job is to close that gap in a day. Not by training a new model — by giving the existing one a kit.

02 The Kit

Four pieces, taught in order: agents, skills, instructions, templates.

These are the primitives HVE Core ships and the ones the workshop drills. Each one is small. Together they replace the "long prompt and a vibe" with named, reusable scaffolding.

Agents

Specialists you summon.

Pre-configured roles — Researcher, Planner, Implementor, Reviewer — that you call by name. Each one has a narrow job and a known output shape, so the next agent in the chain knows exactly what it's reading.

Skills

Knowledge, packaged.

A skill is a folder of domain expertise — a SKILL.md, a few assets, sometimes a script. When a request matches, the agent loads it on demand. You stop pasting the same context into every conversation.

Instructions

Rules the repo enforces.

An .instructions.md file with an applyTo glob. When the model touches a matching file, those rules load automatically. "Use uv for Python," "no --no-verify," "C# tests use xUnit" — written once, applied every time.

Templates

Reusable patterns.

Pre-built scaffolds for the work you do over and over — a new ADR, a PR description, a security review, a sprint plan. The team learns one shape and uses it everywhere.

Collections

Domain bundles.

Pre-curated groups — coding-standards, data-science, security, github, ado. Install the collection your team needs; skip the ones you don't. The workshop helps you pick.

Prompts

Verbs, not essays.

A .prompt.md file becomes a slash-command. "Triage this issue," "review this branch," "draft a PR description." Reusable, parameterized, and version-controlled next to the code they touch.

03 The Loop

The workshop teaches one loop: research, plan, implement.

RPI is the spine of an HVE workflow. Three agents, three artifacts, one human editor — you. The model doesn't decide what to build; it drafts the steps so you can correct course before any code is written.

Research the territory.

The Researcher subagent reads the codebase, the docs, the relevant skills, and any links you drop in. It produces a research note — what exists, what's missing, what's risky — instead of jumping to code.

Plan the change.

The Planner turns the research note into a phased plan with acceptance criteria. You read it like a PR description that hasn't been written yet, push back on it, and only approve when the shape is right.

Implement with a budget.

The Phase Implementor executes one phase at a time, with the plan as its contract. A separate Validator checks the diff against the plan. Drift is caught early, not at review.

The shift isn't that AI writes more code. It's that you stop being surprised by the code it writes. — Workshop debrief, day one
04 What Changes Monday

Before and after the kit lands in the repo.

The workshop's deliverable isn't a slide deck. It's a .github/ folder, a handful of skills, and a team that knows when to reach for which one. The change is visible in the next sprint.

Before the workshop
  • Every prompt re-explains the codebase from scratch.
  • Two engineers ask the same question and get two answers.
  • The model invents a helper that already exists three folders over.
  • "Did the AI write this?" is a question reviewers ask.
  • Security review is a discovery process at the end.
  • Convention drift; no two PRs look alike.
After the workshop
  • Instructions load automatically when the right files are touched.
  • The Researcher cites the file and line before any code is drafted.
  • Skills surface the existing helper instead of inventing a new one.
  • The plan exists in the repo before the diff does.
  • The Security Planner runs as a phase, not as an afterthought.
  • PRs look like each other because the templates wrote them.

None of this requires a new model, a new IDE, or a new license. It requires writing down what your team already knows, and letting the agent system load it on demand.

Closing · The takeaway

AI stops guessing at your repo. It starts working in it.
